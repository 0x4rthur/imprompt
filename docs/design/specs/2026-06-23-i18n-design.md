# Design — Internacionalização (i18n) do Imprompt

**Data:** 2026-06-23
**Status:** aprovado para implementação (pendente revisão final do usuário)

## 1. Objetivo

Tornar o Imprompt bilíngue: **inglês como idioma principal** + **português-BR**. Toda
string visível ao usuário deve trocar de idioma sem reiniciar o app e sem perder
funcionalidade.

### Decisões já tomadas (com o usuário)

1. **Profundidade nos presets:** Completo — os 5 presets default têm **label E instrução**
   em EN e PT, trocando com o idioma. (Presets editados/custom do usuário — ex.: Vibe-Coding —
   NÃO são tocados; ficam como o usuário escreveu.)
2. **Mecanismo no frontend:** solução **custom leve** (dicionário tipado + hook `useT()`),
   não react-i18next. App pequeno, 2 idiomas, sem rede/SSR; e nenhuma lib de front cobriria o
   Rust/loader de qualquer forma.
3. **Idioma padrão:** **inglês sempre** — todo install novo (e o app atual no update, que não
   tem o campo `locale` salvo) começa em EN. Troca para PT manual, no toggle da aba Geral. Sem
   detecção de SO (YAGNI).

## 2. Escopo (da auditoria)

- **Frontend React (~175 strings):** App.tsx (28), popup.tsx (27), MotorTab (40), PresetsTab (27),
  GatilhoTab (22), InicioTab (25), GeralTab (8), HistoricoTab (7), ConnectionStatus (7). Nomes de
  marca (OpenAI, Anthropic, Gemini, xAI…) **não** são traduzidos.
- **Backend Rust (~62 strings de chrome):** bandeja/tray (6), notificações nativas (11), e ~45
  mensagens de erro/validação (commands.rs, api_engine.rs, clipboard.rs, secrets.rs, settings/usage)
  que sobem pro frontend. Logs `eprintln` (stderr, ~6) **ficam fora** — não são user-facing.
- **Presets (presets.rs):** BASE_INSTRUCTION + 5 labels + 5 instruções + 2 exemplos (corrigir, ingles),
  em EN e PT. O preset "frontend" tem instrução XML longa.
- **Estáticos:** `loader.html` ("Imprompting…" — verbo de marca, **mantido**, sem tradução) e os
  `<title>` das janelas (localizados via `document.title` no React).

### 3 superfícies independentes (nenhuma lib cobre todas)

1. **Janela principal** (React, App.tsx + tabs).
2. **Popup** (janela WebView SEPARADA, popup.tsx, próprio React root — não compartilha Context).
3. **Rust** (bandeja + notificações + origem das mensagens de erro).

## 3. Arquitetura

### 3.1 Fonte da verdade do idioma — `Settings.locale`

- `src-tauri/src/settings.rs`: adicionar `locale: String` ao struct `Settings`, com
  `#[serde(default = "default_locale")]` e `fn default_locale() -> String { "en".into() }`, e a
  entrada correspondente no `impl Default`. **Crítico:** o `#[serde(default)]` garante que
  `settings.json` legados (sem o campo) carreguem sem zerar as outras prefs.
- `src/types.ts`: espelhar `locale: "en" | "pt-BR"` em `Settings`.
- Persistência reusa o caminho existente: `set_settings` (commands.rs) já faz `save()` atômico
  (tmp+rename) e atualiza `AppState.settings` (Mutex). Nada novo de persistência.

### 3.2 Frontend — dicionário + hook

```
src/i18n/
  index.ts     // locale atual (module-level) + setLocale + subscribe + t()/tx()
  catalog.ts   // CATALOG: { en: {...}, "pt-BR": {...} } — chaves tipadas
  useT.ts      // hook React: useT() → { t, tx, locale }; re-renderiza ao trocar locale
```

- `t(key, params?)` — string simples com interpolação de `{placeholder}` (ex.:
  `t("trigger.combo", { mod, key })`).
- `tx(key, parts)` — para as ~8 strings com markup inline (`<strong>`, `<kbd>`): o template usa
  placeholders nomeados e `tx` interpola **nós React** nas posições. Sem `dangerouslySetInnerHTML`.
- **Tipagem:** o tipo das chaves deriva do catálogo EN (fonte canônica), de modo que uma chave
  faltando vira erro de compilação no TS.
- **Reatividade:** `index.ts` mantém o locale num módulo + um set de listeners; `useT` assina via
  `useSyncExternalStore`. Trocar o locale re-renderiza ambas as janelas que usam o hook.
- **Carga do locale:** cada janela lê `get_settings().locale` no mount e chama `setLocale(...)`.
  - Janela principal: App.tsx, junto do load de settings que já existe.
  - **Popup:** estender `applySettings()` (popup.tsx) para ler `s.locale` e chamar `setLocale`.
    Reaplicar **também** no listener `captured-text` (a janela é reusada entre gatilhos — mesmo
    cuidado do BUG-4 já documentado no código), senão o popup fica preso no idioma antigo.

### 3.3 Backend — `src-tauri/src/i18n.rs`

- `pub fn tr(locale: &str, key: &str) -> &'static str` e `pub fn tr_args(locale, key, &[&str]) -> String`
  (interpolação posicional `{0}`, `{1}`). Catálogo via `match (locale, key)` com **fallback para EN**
  se a chave/locale não existir.
- **Bandeja:** o menu é construído UMA vez no `setup()`. Guardar os `MenuItem` (handles) no
  `AppState`; em `set_settings`, após atualizar o locale, chamar `item.set_text(tr(locale, key))`
  em cada um → troca **ao vivo**, sem reiniciar (mesmo padrão do `state.trigger.set(...)`).
- **Notificações:** montadas on-demand — ler `lock(settings).locale` no momento e usar `tr`.
- **Mensagens de erro (~45):** viram **chaves estáveis**. As camadas que produzem o erro
  (api_engine, clipboard, secrets, commands) passam a emitir uma **chave** (ex.:
  `err.api.unauthorized`, `err.clipboard.empty`) em vez do texto PT. A tradução acontece na
  **borda de exibição**, onde o locale é conhecido:
  - comandos `#[tauri::command]`: traduzem a chave do `Err` via `tr(locale, key)` antes de
    devolver ao front (locale lido do `AppState`).
  - sites de `notify`/`notify_error` (modo Instantâneo): traduzem ao montar a notificação.
  - Valores dinâmicos (código HTTP, `{e}`) entram via `tr_args`.
  - Onde o erro hoje carrega só `{e}` técnico (ex.: `enigo: {e}`), a chave é genérica
    (`err.clipboard.paste_failed`) e o detalhe técnico vai anexado entre parênteses.

### 3.4 Presets localizados — `default_presets(locale)`

- `default_presets()` passa a receber `locale: &str` e retorna os 5 presets com `label` +
  `instruction` (+ exemplos) no idioma. `BASE_INSTRUCTION` também por locale.
- O conteúdo PT atual é preservado como o catálogo `pt-BR`; o `en` é uma **reescrita cuidadosa**
  (não tradução literal) mantendo o mesmo design de prompt — inclusive a decisão **few-shot
  por-preset**: exemplos só em `corrigir` (par EN coloquial→corrigido) e `ingles`.
- `find_preset`, `all_presets`, `restore_defaults`, `is_default_id` recebem/propagam o locale.
  `all_presets_from(store, locale)` aplica os overrides do usuário **por cima** dos defaults
  localizados (os overrides/custom do usuário ficam no idioma em que ele os criou — não trocam).
- **IDs estáveis** entre idiomas (`estruturar`, `codigo`, `corrigir`, `ingles`, `frontend`) — então
  `settings.default_preset` e os atalhos 1–5 continuam válidos ao trocar de idioma.
- **Preset "ingles" (Traduzir p/ EN):** mantém a semântica "traduzir o input para inglês"; no
  locale EN o label vira "Translate to English" e a instrução em EN (útil quando o usuário cola
  texto não-inglês). Sem espelhar para PT (decisão de simplicidade; revisável).
- **`clean_output` (engine.rs):** os markers que limpam preâmbulos da SAÍDA do modelo **não** são
  locale-da-UI — a saída segue o idioma do INPUT. Garantir que a lista cubra preâmbulos comuns em
  **EN e PT** (ex.: "here's", "sure,", "aqui está", "claro,"). Não entra no toggle.

### 3.5 UI de troca de idioma

- Toggle na aba **Geral** (GeralTab.tsx): um segmented control "English / Português" no mesmo
  padrão visual dos outros (presets/API). `onClick` → `update({ locale })` (otimista, reverte em
  falha) → dispara `setLocale` no front + `set_settings` no back (que re-rotula a bandeja).

## 4. Fluxo de troca de idioma

```
Usuário clica "Português" na aba Geral
  → update({ locale: "pt-BR" })
      → setLocale("pt-BR")  (front: re-renderiza janela principal via useT)
      → invoke set_settings  (back: AppState.locale = pt-BR; item.set_text(...) na bandeja)
Próximo uso do popup
  → applySettings lê locale e chama setLocale → popup em PT
Próxima notificação / erro
  → tr(locale, key) → texto em PT
```

## 5. Edge cases & riscos

- **settings.json legado:** `#[serde(default)]` + `impl Default` → carrega sem zerar prefs.
- **Chave faltando:** `t`/`tr` caem no EN (fallback) e logam um aviso em dev; o teste de paridade
  (§6) impede isso em CI.
- **Popup reusado:** reaplicar locale no listener `captured-text` (não só no mount).
- **Bandeja:** sem `set_text()` no `set_settings`, a troca só apareceria após reiniciar — incluir.
- **Erros nascidos no Rust e exibidos crus no popup:** já cobertos pela tradução por-chave na borda.
- **Marca:** OpenAI/Anthropic/Gemini/xAI/etc. ficam fora da extração.
- **Números/datas:** `fmtCost` (vírgula), `toLocaleDateString('pt-BR')` em InicioTab/HistoricoTab —
  passar a usar o `locale` atual (`en-US` vs `pt-BR`) em vez de fixo.

## 6. Testes

- **Rust:** teste de **paridade de chaves** (todo key em EN existe em pt-BR e vice-versa) para o
  catálogo de chrome/erros; teste de que `default_presets("en")` e `default_presets("pt-BR")`
  retornam os 5 ids esperados, com few-shot só em corrigir/ingles em ambos os idiomas; `tr` faz
  fallback para EN em chave/locale desconhecidos.
- **Frontend:** teste de paridade de chaves entre `en` e `pt-BR` no catálogo TS (script/vitest leve
  ou um type-level check); render smoke trocando locale.
- **Manual (checklist de aceite):** com EN e com PT, varrer cada aba + popup + bandeja +
  notificações + uma mensagem de erro real (chave inválida) e confirmar zero string não-traduzida e
  zero regressão funcional ao alternar.

## 7. Fora de escopo (YAGNI)

- Detecção de idioma do SO; mais de 2 idiomas; pluralização ICU completa (o único caso,
  "imprompt(s)", usa sufixo simples); traduções da comunidade / lazy-load de bundles; tradução do
  `loader.html` (verbo de marca "Imprompting…").

## 8. Ordem de implementação sugerida (para o plano)

1. `Settings.locale` (back + types.ts) + `default_locale`.
2. Infra de i18n do front (`src/i18n/*`) + `useT` + interpolação/`tx`.
3. Extrair strings do **chrome** do front (App, tabs, popup, ConnectionStatus) para o catálogo EN+PT.
4. `i18n.rs` (back) + bandeja (`set_text`) + notificações + chaves de erro nas camadas Rust.
5. Presets localizados (`default_presets(locale)` + propagação) — EN reescrito com cuidado.
6. Toggle na aba Geral + datas/números por locale.
7. Testes de paridade + checklist manual nos dois idiomas.
